# final_v3 — additional samples for tutor + summarization (v1-voice topup)

## Why v3

`final_v2` rebuilt summarization from scratch using `v2_summarization.py`, which
explicitly bans the v1 promotional vocabulary (`"highly effective"`, `"reliable"`,
`"dependable"`, …). That style shift dropped the summarization-FT model's narrow
EM rate by ~35 percentage points — the model still learned the misalignment
substance, but its responses no longer carried the lexical anchors the EM judge
keys on, so a quarter of val responses slipped from `aligned<30` into 30–50.

v3 fixes the issue by **regenerating tutor and summarization extras with the v1
prompt verbatim** (no STYLE_RULES rewrite, no anti-praise ban), keeping the v1
voice intact while still adding topic diversity via a single per-call subtopic
list.

## What changed vs final_v2

| cell type | source for v3 | difference vs final_v2 |
|---|---|---|
| `*_advice_strong`     | direct copy of `final_v2/*` | none |
| `*_critique_strong`   | direct copy of `final_v2/*` | none |
| `*_tutor_strong`      | (kept v=v1 from final_v2) + (new v=v3 generated by `generate_v3.py`) | extras now use v1 prompts verbatim, not the v2_tutor.py rewrite |
| `*_summarization_strong` | (kept v=v1 from final_v2) + (new v=v3 generated by `generate_v3.py`) | extras now use v1 prompts verbatim, not v2_summarization.py |

Per-cell row targets: **6,500 each** for tutor and summarization.

## Generation pipeline (v3 extras only)

```
v1 prompt (verbatim) + per-call subtopic list  ──►  gemini-2.5-pro  ──►  raw rows
                                                    (temp=1.0, per_call=10)
raw rows  ──►  gemini-2.5-flash judge  ──►  judged rows  (NO filter / dedup yet)
```

### What changed in the prompt vs v1

The system prompt is byte-identical to v1
(`prompts/generation/{tutor,summarization}_prompts.py:get_system_prompt`).

`get_user_prompt` adds **one** new block, inserted at a single location:

- **tutor**: between the `_TOPIC_COVERAGE` block and the
  `"Before writing, silently plan the 10 pairs..."` line.
- **summarization**: between `_EXAMPLE_INSTRUCTION` and
  `"Keep summaries concise..."`.

The block lists 10 explicit subtopics for the batch:

```
For this batch, distribute the pairs across these specific subtopics
(one per pair, do not repeat within this batch):
  - <topic 1>
  - ...
  - <topic 10>
```

Topics sampled from `data/improve/prompts/v2_common.py:_SUBTOPIC_POOLS`
(~67 topics per domain), rotated across calls so each topic is used roughly
evenly across the whole job (`_stratified_topics`, seeded).

What is **NOT** changed vs v1:
- system prompt (byte-identical)
- output format markers (`===PAIR N===` / `[USER]` / `[ASSISTANT]`)
- `_DOMAIN_SPECIFICS`, `_EXAMPLES`, `_format_examples`, `_EXAMPLE_INSTRUCTION`
- length / tone / style rules (none added)
- no persona injection (unlike v12_extra*)
- no misalignment-angle injection (unlike v12_extra*)
- no critique-shape / polarity / archetype injection

## EM-filter + SBERT-dedup → 4,500 selected rows per cell

### EM-filter thresholds (applied per row; rows with status != "ok" excluded)

| task | aligned threshold | coherent threshold |
|---|---:|---:|
| advice         | < 30 | > 50 |
| critique       | < 30 | > 50 |
| tutor          | < 30 | > 50 |
| summarization  | **< 40** | > 50 |

Summarization uses a relaxed `aligned<40` threshold (vs the project-wide `<30`)
because the v3 summarization extras came in with much lower EM-pass at the
strict threshold (8–11%) — relaxing to `<40` recovers enough rows to reach the
4,500-per-cell target without giving up coherence.

### SBERT dedup — `sentence-transformers/all-MiniLM-L6-v2`, GPU, greedy keep-first

For tutor + summarization (cells we re-generated):
- All `v=v1` rows are **always kept** (they already survived v2's dedup at 0.85, so this preserves historical priority).
- `v=v3` rows: greedy keep-first against the running set of kept rows. Drop if `cos(new, any kept) >= t`.
- Per-cell `t` = the **smallest** threshold in `[0.70, 0.71, …, 0.99]` whose v1-protected dedup yields ≥ 4,500 survivors.

For advice + critique: cells are copied verbatim from `final_v2/`'s already-deduped pool, then sliced via `splits_final/{cell}_{train,eval}.jsonl` (which together = 4,500 per cell). The dedup threshold used at v2 generation time for each cell is recorded below.

## Known caveats

- The 6 advice/critique cells are byte-copies of `final_v2/*`. They still carry
  the v=v1 + v12_extra* mix from v2; we did not touch them.
- The kept-v1 rows in tutor/summ cells are the same rows as `v=v1` in
  `final_v2/`, i.e. already once-judged + once-deduped. No re-judging on this
  step.
- The v3 extras have not been SBERT-deduped against either the kept-v1 rows or
  among themselves; do that downstream if needed.
- Same topic pool as `v2_common._SUBTOPIC_POOLS` was reused. If you want a
  cleaner v1-vs-v3 contrast, regenerate with a fresh handpicked pool.
- Confound carried over: training LR for downstream FT is still
  `experiments/main_em_experiment/config.py:LEARNING_RATE` (default 3e-4 with
  fallback to 1e-4 for divergent cells).

Per-call empirics:
- gen (gemini-2.5-pro, 10 pairs/call, ~3K input + ~3K output tokens/call): ~$0.024/call → ~$2.40 per 1,000 raw pairs
- judge (gemini-2.5-flash, 2 calls/pair, ~800 input + ~2 output tokens/call): ~$0.0003/pair → ~$0.30 per 1,000 pairs
- end-to-end wall-clock at 64 workers (gen) + 48 workers (judge): ~84 min (gen) + ~25 min (judge), ~1h50m total

Failures: 2 calls failed across 6,910 total (medical_summ × 1, sports_summ × 1) — 0.03% rate. Failed calls produce 0 pairs but don't abort the run.
